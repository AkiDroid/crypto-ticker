## Context

当前仓库还是一个新项目，目标产品是一个运行在 macOS 状态栏中的加密货币价格查看应用。当前阶段只需要完成项目初始化和整体框架搭建，不实现真实 API 请求、定时刷新、币种切换或生产级数据展示逻辑。

该设计需要同时满足两个约束：一是应用在初始化后应当具备清晰的状态栏产品形态，便于后续直接接入业务；二是工程结构必须为未来的价格服务、配置管理和测试扩展预留稳定边界，避免把早期原型代码固化为长期架构。

## Goals / Non-Goals

**Goals:**
- 建立一个可编译的 macOS 状态栏应用骨架，包含应用入口、状态栏项、基础菜单和占位展示。
- 明确 UI 层、应用编排层、领域模型层、基础服务层之间的职责边界。
- 定义未来价格拉取、刷新调度和配置能力所需的协议、模型和占位实现。
- 初始化测试 target 与最小可运行的测试样例，确保工程从一开始就具备可验证性。

**Non-Goals:**
- 不实现任何真实的加密货币价格 API 调用。
- 不实现定时刷新、缓存、错误重试、用户配置持久化等业务逻辑。
- 不引入第三方网络、数据库或响应式框架，除非初始化工程本身无法完成。
- 不处理打包发布、签名、公证或自动更新流程。

## Decisions

### 1. 使用原生 macOS 技术栈搭建首版骨架
选择使用 Swift、SwiftUI 和 AppKit 状态栏能力完成应用壳结构，而不是一开始引入跨平台框架。

原因：
- 当前目标明确限定为 macOS app，原生栈能最直接支持状态栏交互。
- 状态栏应用与应用生命周期、菜单、弹出交互高度绑定，原生 API 更稳定。
- 在无业务逻辑阶段优先降低技术复杂度，避免为未来未确认需求支付额外抽象成本。

备选方案：
- 使用 Electron 或 Tauri 等跨平台方案。
未采用原因：
- 会显著增加工程体积、依赖复杂度和初始化成本，与当前“先搭骨架”的目标不匹配。

### 2. 采用分层结构而不是单文件原型
工程按 `App`、`Features`、`Domain`、`Services`、`Support`、`Tests` 等职责拆分，而不是把状态栏 UI 与未来服务逻辑混写在同一层。

原因：
- 状态栏展示、价格服务、刷新调度和配置读取在后续会快速演进，分层可以降低耦合。
- 先定义协议和占位实现，可以让后续业务功能以替换实现的方式推进，而不是大规模重构入口代码。

备选方案：
- 先用单目录或单文件做最小演示，再后续重构。
未采用原因：
- 当前用户明确要求“初始化项目并创建整个项目框架”，单文件原型不满足这个交付目标。

### 3. 为未来的数据接入定义协议和 stub，而不提前写假业务流程
为价格提供者、刷新调度器、应用配置等定义协议与默认 stub，实现空数据或占位数据返回；业务流程保持未接入状态。

原因：
- 这样既能让工程结构完整，又不会误导后续实现依赖错误的假逻辑。
- stub 便于 UI 和应用编排层在没有真实 API 的情况下完成集成。

备选方案：
- 直接实现一个调用公开 API 的临时版本。
未采用原因：
- 与当前“先不要实现具体业务功能”的要求冲突，也会过早引入外部依赖和错误处理路径。

### 4. 测试先覆盖结构性契约，而不是业务正确性
初始化阶段的测试只验证核心组件可实例化、占位服务可调用、状态栏骨架的关键入口存在。

原因：
- 当前没有真实业务行为，测试目标应该放在工程边界和依赖注入约束上。
- 这样可以为后续规格演进提供稳定测试基线。

备选方案：
- 暂不建立测试 target，等功能实现后再补。
未采用原因：
- 会让项目骨架从第一天开始缺少验证入口，不利于后续迭代。

## Risks / Trade-offs

- [状态栏 UI 依赖 AppKit/SwiftUI 桥接细节] -> 通过在骨架阶段收敛到最小交互面，只暴露必要的应用编排接口来降低后续改动面。
- [过早抽象服务边界可能导致命名或职责调整] -> 协议只保留最小必要接口，避免在设计阶段定义过多未来字段。
- [无真实 API 时占位实现可能让 UI 看起来“完成度过高”] -> 在占位文案和 spec 中明确声明仅为 skeleton，不承诺生产行为。
- [初始目录结构可能与后续团队偏好不完全一致] -> 通过 feature 与 domain 分离，使后续重组成本保持在目录级而非架构级。

## Migration Plan

1. 初始化 macOS 应用工程和测试 target。
2. 建立状态栏应用入口、菜单和占位展示，确保应用可以以骨架形态运行。
3. 加入领域模型、协议和 stub 服务，打通依赖注入链路但不实现真实业务。
4. 增加最小测试样例和基础文档。
5. 后续功能开发以替换 stub、补全实现和扩展测试为主，不需要推翻骨架。

当前为新项目初始化，不涉及线上迁移与回滚；如骨架方向需要调整，可直接在实现阶段重构工程文件与模块目录。

## Open Questions

- 状态栏点击后最终采用纯菜单交互、popover，还是独立窗口入口，当前只需保留可替换骨架。
- 后续价格 API 是否需要多提供商支持，这会影响价格服务协议的抽象粒度。
- 项目初始化是否需要同时配置 CI、lint 或格式化工具，当前变更暂不强制要求。
