# AGENTS.md

本仓库不再使用 OpenSpec。后续请使用下面这套轻量文档约定，方便 Codex 和其他 AI 编程工具快速理解项目并持续维护。

## 输出要求

- 始终使用中文输出。
- 修改功能代码后，必须同步更新文档；不要只改代码不改文档。

## 文档入口

开始任何实现前，优先按这个顺序读取：

1. `README.md`
2. `docs/PROJECT.md`
3. `docs/CHANGELOG.md`

## 轻量工作流

1. 先阅读当前文档，确认项目现状和边界。
2. 实现功能或修复问题。
3. 根据改动同步更新文档。
4. 在回复中明确说明改了哪些代码文件、改了哪些文档文件。

## 文档更新规则

### 必改：`docs/CHANGELOG.md`

只要有实际代码改动，就追加一条记录，至少包含：

- 日期
- 变更标题
- 背景 / 目的
- 代码改动点
- 文档同步情况
- 验证情况

### 按需更新：`docs/PROJECT.md`

当以下内容发生变化时，必须更新：

- 当前能力范围
- 目录结构
- 架构边界
- 关键依赖和约束
- 运行或测试方式

### 按需更新：`README.md`

当以下内容发生变化时，必须更新：

- 项目简介
- 使用方式
- 快速开始
- 对外可见能力

## 变更粒度约定

- 小型修复：可以只更新 `docs/CHANGELOG.md`
- 新功能或行为变化：更新 `docs/CHANGELOG.md`，并视情况更新 `docs/PROJECT.md` / `README.md`
- 架构调整：三份文档都要检查是否需要更新

## 禁止事项

- 不要重新引入 OpenSpec 的 proposal/design/tasks/spec 流程
- 不要新增与当前代码状态不一致的规划性文档
- 不要把“未来可能做”的内容写成“当前已经支持”

## 记录原则

- 文档只写当前事实、稳定约束、明确的后续协作方式
- 若实现与旧文档冲突，以当前代码为准，并立刻修正文档
